Why your SAST keeps missing business-logic flaws

SAST is excellent at finding patterns it has been taught. Business logic is, by definition, the part it has not. Here is how to fill the gap without doubling your AppSec spend.

Priya AnandChief Threat Officer

Pentests keep finding the same class of issues that your SAST never raises. Coupon codes that stack against the rules. Refund flows that issue value before the source charge clears. Step-up authentication that can be skipped by replaying an earlier session. SAST is not broken. It is being asked to do something it cannot do.

What SAST does well

Pattern matching against known dangerous primitives. SQL string concatenation. Unvalidated deserialization. Hard-coded credentials. Insecure defaults in popular libraries. For these, modern SAST is fast, cheap, and accurate.

Where it falls down

Business-logic flaws are not patterns. They are violations of intent. The code compiles, the tests pass, the security scanner sees nothing wrong — but the flow does something the business never meant to allow. Three examples we have all seen:

  • A discount code that is checked but not invalidated, allowing reuse
  • A multi-step form that lets you submit step three without completing step two
  • An API that accepts a userId in the body and trusts it instead of the session

None of those flaws have a syntactic signature. SAST cannot see them.

Business logic vulnerabilities are not bugs in the code. They are mismatches between what the code does and what the business meant for it to do.

What actually catches them

Threat modeling, before the build

A 90-minute threat model on a feature the size of a checkout flow will surface more business-logic risk than any scanner. The exercise is cheap; the discipline is the hard part.

Abuse cases in your test suite

For every "happy path" automated test, write at least one "abuse path." What if the user skips a step? What if they reuse a token? What if they submit a value the form was never supposed to send? These belong in the same test runner as your normal tests, not in a separate security tool.

Manual code review focused on flows, not files

Read the code following the user's journey. Sign up, log in, place an order, request a refund. You will find issues a file-by-file review never would, because the flaws live in the seams between modules.

Targeted penetration testing

Scope a quarterly pentest specifically against business-logic abuse. Give the testers a feature, not a URL list. The findings will tell you which flows need attention next.

What to do this quarter

Pick one critical flow — the one that, if abused, would cost you the most. Run a threat model on it. Write three abuse-case tests against it. Have a senior engineer read the flow end to end. Then move to the next one. You will catch business-logic flaws faster than any tool can teach itself to find them.

SAST is a useful baseline. It is not an AppSec program. The flaws that hurt the most are the ones that look like working software.
