Hardening AWS IAM: 12 rules we apply to every environment

IAM is where most cloud breaches start and most cloud audits stall. These twelve rules are the baseline we deploy on day one of every engagement.

Erik LindqvistCEO & Co-Founder

If you only fix one layer of your AWS estate, fix IAM. The vast majority of incidents we respond to in cloud environments trace back to over-permissive policies, dormant access keys, or weak conditions on assume-role trust. The good news: a tight IAM baseline is achievable in a single sprint.

Identity hygiene

  1. No long-lived access keys for humans. All console access via SSO with MFA. Programmatic access via short-lived credentials.
  2. Disable root account access keys. Use the root account only for the few operations that require it.
  3. Rotate service-account keys quarterly. Use AWS Secrets Manager or a workload identity solution to make rotation a non-event.
  4. Block creation of IAM users. Service control policy at the org level. New principals come from your IdP, not from individual accounts.

Policy quality

  1. No "Action": "*" in customer-managed policies. If a workload truly needs broad access, scope it with conditions.
  2. Use aws:ResourceTag and aws:PrincipalTag conditions. Tag-based access scales better than enumerated ARNs.
  3. Boundary policies on every privileged role. A permission boundary is the cheapest way to say "this role can never escalate beyond this set."
  4. Deny conditional public S3 access at the bucket and account level. Block Public Access at the org account; deny "aws:PrincipalIsAWSService": "false" on internal buckets.
A wildcard action with a tight condition is fine. A wildcard action with no condition is a finding.

Cross-account and federation

  1. External ID required on every cross-account assume-role trust. No exceptions. Generate per-vendor.
  2. Source-IP conditions on break-glass roles. Limit emergency-access roles to your VPN range or a known bastion.
  3. Session duration capped at one hour for production roles. Long sessions extend the window of stolen-credential abuse.

Detection

  1. CloudTrail organization trail with log-file integrity. Lake or Athena for queries. Detection rules for iam:CreateAccessKey, iam:AttachUserPolicy, sts:AssumeRole from new IPs, and any use of root.

What to do this week

If you cannot fix all twelve in one go, fix these three first: disable root access keys, deny new IAM user creation org-wide, and audit every assume-role trust for missing external IDs. Those three close the largest blast-radius issues we see in incident response.

IAM is one of the rare areas in cloud security where the baseline is well understood and the hard part is execution discipline. Start now, not after the audit.
